2. SHORT PRESENTATION SCRIPT
==================================================

Hello everyone. My project is an interactive dashboard on New York City food safety using restaurant inspection data.

The main driving question of my project is:
How do food safety outcomes vary across restaurants, boroughs, cuisines, and inspection patterns in New York City, and how can those patterns help users make safer dining decisions?

I started with the NYC restaurant inspection dataset and first worked on cleaning and organizing the raw data. After that, I engineered additional features such as risk level, time-based fields, and simplified violation categories so the data would be easier to analyze and explain.

I first explored the project in a notebook and then converted it into a Dash application so the result could be shown as an interactive dashboard in the browser. The dashboard includes summary KPIs, borough comparison, time trends, cuisine-level analysis, violation category summaries, an interactive restaurant map, and a detail panel for selected restaurants.

The target audience includes students, diners, tourists, and a classroom audience interested in how data can be cleaned, structured, and turned into an interactive decision-support tool.

The goal of the project is not just to show charts, but to help users move from a broad citywide view to a more specific restaurant-level understanding of food safety patterns.

4. HOW TO ANSWER THE FINAL PROJECT GOALS
==================================================

GOAL 1
Provides an overview that clearly states the driving question and links the question to the data approach

How to answer:
My driving question is about identifying patterns in NYC restaurant food safety and using those patterns to support safer dining decisions. I linked this question to the data approach by using inspection-level records from the NYC restaurant inspection dataset, cleaning them, and organizing them into views that support borough comparison, cuisine comparison, time trends, and restaurant-level map exploration.


GOAL 2
Describe the data sources, quality, and structure used in the project

How to answer:
The primary source is the NYC restaurant inspection dataset. The data includes restaurant identifiers, business names, borough, cuisine type, inspection dates, grades, scores, violation descriptions, and geographic coordinates. In terms of quality, the raw data contained missing values, invalid placeholders such as borough value 0, repeated inspection records, and coordinates that were not always usable for mapping. Structurally, the data is inspection-based, meaning that one restaurant can appear multiple times across different inspections, so I had to create both a historical inspection view and a latest-inspection restaurant snapshot.


GOAL 3
Documents the collection, wrangling, and analysis, and development of an interactive data visualization

How to answer:
The project documents the full process from raw data to interactive output. I standardized columns, fixed data types, handled missing and invalid values, removed duplicates, created derived variables such as risk level and simplified violation categories, and then developed an interactive Dash application to present the final insights. The notebook captures the exploratory and milestone analysis, while the Dash app represents the final interactive visualization layer.


GOAL 4
Accurately applies course concepts for organizing and analyzing data and for visually presenting data

How to answer:
This project applies course concepts by organizing messy data into structured analysis-ready datasets, creating meaningful derived variables, and using appropriate chart types for different analytical questions. It also applies visual presentation concepts by aligning visual encoding with meaning, such as risk-based color logic, geographic mapping, filtering, and summary-to-detail interaction design.


GOAL 5
Prioritizes the audience’s needs and priorities in the design and development decisions

How to answer:
The design decisions were made with both an academic and practical audience in mind. For a classroom audience, the dashboard provides clear summary views, trends, and comparisons. For a practical audience such as students, tourists, or diners, the map and safer-picks table provide actionable information. The filters, simplified violation categories, and click-based detail view were all chosen to make the dashboard easier to use and easier to interpret.


GOAL 6
Reflects on limitations of the final project and discusses alternative approaches and next steps for the future

How to answer:
One limitation is that the dataset is observational and based on inspection records, so it does not capture every possible dimension of restaurant safety. Another limitation is that some source data fields were missing, invalid, or inconsistently recorded, which required cleaning decisions such as labeling some values as not reported. The violation categorization is also a simplified grouping rather than a full natural-language analysis. For future work, I could improve the project by adding borough-over-time comparisons, more advanced text analysis of violations, or a deployed public version of the dashboard.

8. LIKELY QUESTIONS FROM CLASSMATES OR PROFESSOR
==================================================

Q: Why did you use Dash instead of only a notebook?
A: A notebook is useful for exploration, but Dash is better for presenting an interactive final product. It lets the audience interact with the data directly through filters and map exploration.

Q: Why did the restaurant count change during development?
A: The raw dataset had over 30,000 unique restaurant IDs, but the dashboard uses a cleaned version that removes unusable inspection rows and duplicates. Earlier versions also applied a stricter grade filter, which reduced the count further.

Q: What does “records” mean in the violation chart?
A: Records means inspection rows, not unique restaurants. A restaurant can appear multiple times because it may be inspected more than once.

Q: What does “Not Reported” mean?
A: It means the source dataset did not provide a valid value for that field, or the field contained an invalid placeholder such as 0 in the borough column.

Q: Why do you use the latest inspection for some charts?
A: Some charts are meant to compare restaurants in their most current state, so using the latest inspection avoids overcounting restaurants with many historical inspections.

Q: Why is the map useful if other charts already exist?
A: The map adds a spatial dimension. It helps users connect inspection outcomes to geography and supports more practical exploration.
